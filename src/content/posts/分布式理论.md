# 什么是分布式系统

多台通过网络连接的独立计算机，为了完成共同的任务而协同工作，但在用户看来是一个单一、连贯的系统。

# 特征

## 分步性

节点可以位于同一个机房，也可以分散在全球各地。虽然物理上分散，但它们在逻辑上是一个整体，共同对外提供服务。

## 透明性（核心）

用户（或上层应用）**感觉不到**系统是分布式的。用户不需要知道数据存在哪台机器上、任务是由哪个节点处理的、网络是如何路由的。就像你使用 Google 搜索时，并不需要知道你的请求被发到了哪个数据中心。

## 并发性

多个节点可以同时处理不同的任务，或者共同处理同一个大任务（并行计算）。极大地提高了系统的吞吐量和处理速度。

## **对等性/自治性**

自治：每个节点都有自己的处理器、内存和操作系统，能独立运行。

对等：在很多架构中，节点之间地位平等（如 P2P 网络），没有绝对的中心控制点；即使有主从架构（如 Master-Slave），从节点也具备独立处理能力，且主节点故障时可切换。

## **缺乏全局时钟与故障常态性**

没有统一时间：由于网络延迟，很难在所有节点上维持一个精确的全局时间序列（需要靠 NTP 或逻辑时钟算法解决）。

故障是常态：在大规模集群中，部分节点宕机、网络断连是**随时可能发生**的。分布式系统的设计前提就是**假设硬件不可靠**，并通过软件机制（如冗余、重试、容错）来保证系统整体依然可用。

# 为什么要分布式系统呢？

1. **突破单机性能瓶颈**：单台服务器的 CPU、内存、磁盘 IO 都有上限。当数据量达到 PB 级或并发量达到百万级时，单机根本扛不住。分布式可以通过增加机器线性提升能力。
2. **高可用性** (High Availability)：**分布式系统中的每一个独立服务，都通过集群方式部署，确保单个服务不挂**，即使某台服务器甚至整个机房断电，其他节点也能立刻接管工作，保证服务不中断（如银行系统、电商双11）。
3. **高可扩展性** (Scalability)：**水平扩展**：业务增长时，只需简单地“加机器”即可，无需更换更昂贵的大型机。
4. **地理就近服务**：将数据和服务部署在离用户最近的节点（CDN、边缘计算），降低网络延迟，提升用户体验。

# 分布式系统和微服务架构

这个应该会很容易搞混，因为看上去很像，并且经常一起出现。

分布式系统是<u>***物理/架构***</u>形态概念。

微服务是一个<u>***逻辑/应用***</u>设计概念。

<u>*比如分布式要在不同机器上运行，微服务可以部署在同一个机器。*</u>

为了更好的区分，以下是通过千问生成的对比表，各位可以看看

| 维度     | 分布式系统 (Distributed)                      | 微服务架构 (Microservices)                          |
| -------- | --------------------------------------------- | --------------------------------------------------- |
| 本质属性 | 基础设施/部署模式                             | 应用架构/设计风格                                   |
| 核心目标 | 高性能、高可用、大容量 (让系统跑得快、不挂掉) | 高内聚、低耦合、敏捷开发 (让代码好写、好改、好上线) |
| 拆分依据 | 按资源拆分 (如：分库分表、读写分离、集群扩容) | 按业务领域拆分 (如：用户域、订单域、支付域 - DDD)   |
| 通信方式 | 可能是内部共享内存，也可能是网络 RPC          | 必须是网络通信 (HTTP, gRPC, MQ)                     |
| 数据管理 | 通常共享一个大数据库 (Sharding)               | 数据库私有化 (每个服务独占自己的DB)                 |
| 依赖关系 | 节点间通常是对等的或主从的，功能相同          | 服务间存在复杂的调用链依赖                          |
| 典型场景 | 大数据计算 (Hadoop)、搜索引擎、CDN            | 复杂业务系统的快速迭代、多团队并行开发              |

# CAP理论

既然讲到分布式系统，就不得不讲一下CAP理论。它指的是分布式系统中<u>***Consistency（一致性）***</u>、<u>***Availability（可用性）***</u>、<u>***Partition tolerance（分区容错性）***</u>，三者不可得兼

## 一致性

在分布式系统中的所有数据备份，在同一时刻是否同样的值（等同于所有节点访问同一份最新的数据副本）

## 可用性

在集群中一部分节点故障后，集群整体是否还能响应客户端的读写请求（对数据更新具备可用性）

## 分区容忍性

以实际效果而言，分区相当于对通信的时限要求，系统如果不能在时限内达成数据一致性就意味着发生了分区的情况，必须就当前操作在C（一致性）和A（可用性）之间做出选择。

<u>*PS：比如有一个集群，节点 A 在北京机房，节点 B 在上海机房。北京和上海之间的光缆被挖断了，或者交换机故障了。对于节点 A 来说，它不知道节点 B 是死了，还是只是网络不通；对于节点 B 也一样。这就形成了一个**网络分区**。整个系统被割裂成了两个独立的孤岛。*</u>

# BASE理论

它为了解决 **CAP 理论**中“一致性（C）”和“可用性（A）”难以兼得的矛盾，而提出的一种**柔性事务**设计理念。

它的核心思想是：既然强一致性太难做到，也会拖垮系统性能，那我们能不能妥协一下？允许数据在短时间不一致，只要保证最后的结果是对的就好了，以此来换取系统极高的可用性。

分为**基本可用（BA）**、**软状态（S）**、**最终一致性（E）**

## **Basically Available（基本可用）**

分布式系统在出现故障（如机房断电、网络拥堵、部分节点宕机）时，**允许损失部分可用性**，但必须保证**核心功能**依然可用。举个例子：双十一，为了保证核心功能，会把查看商品历史评价、申请退款这类非核心功能暂时停掉。

## **Soft state（软状态）**

允许系统中的数据存在**中间状态**，并认为该中间状态的存在不会影响系统的整体可用性。

传统数据库（ACID）要求**硬状态**：事务一旦提交，数据必须立即一致，不能有“正在同步中”的状态。

BASE 允许**软状态**：数据可以在不同节点之间存在短暂的不一致，系统内部可以有“正在处理”、“同步中”这样的中间态。

## **Eventually consistent（最终一致性）**

这是 BASE 理论的**终极目标**。虽然系统在过程中允许数据不一致（软状态），但经过一段时间后（网络恢复、后台补偿机制执行完毕），所有节点的数据**最终会达到一致**。
