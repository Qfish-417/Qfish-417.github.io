# 索引

这大概是学习数据库的时候都得要知道的一个知识点，现在为了方便自己学习就顺便把这个知识点整理一下

# 为什么需要索引？

这就相当于查字典的时候，可以通过一定的规律查找一个字，否则你就需要O(n)查找。当数据及其庞大的时候，O(n)查找这是非常影响效率的

# 分类

以下是按照不同角度进行的分类

## 数据结构

### B+tree索引

简单来说，它是一棵**“矮胖”且“所有数据都在最底层”**的多路平衡查找树。它的设计初衷是为了**最大化减少磁盘 I/O 次数**，从而在海量数据下实现极速查询。它有**<u>叶子节点</u>**和**<u>非叶子节点</u>**。其中非叶子节点只存索引，不存数据，只是纯粹的路标。

### Hash索引

它的核心思想是：**通过哈希函数将索引键（Key）直接映射到内存中的某个具体位置（桶/Bucket），从而实现 O(1) 时间复杂度的极速查找。**

### Full-text索引

**在海量文本中，快速找到包含特定“关键词”或“短语”的记录，并能按相关性排序。**

## 物理存储

### 聚簇索引（主键索引）

聚簇索引是一种**将数据存储与索引放到一块**的索引结构。在 InnoDB 中，**数据行本身就是索引的叶子节点**。

### 二级索引（辅助索引）

二级索引是除了聚簇索引之外的其他索引。它的叶子节点**不存储完整行数据**，只存储**索引列的值**和**对应的主键值**。

## 字段特性

### 主键索引

一种特殊的**唯一索引**，用于唯一标识表中的每一行记录。

### 唯一索引

要求索引列的值在整张表中必须**唯一**，但**允许为 NULL**（注意：在大多数数据库中，NULL 被视为不相等，所以可以有多个 NULL 值，具体视版本而定，MySQL InnoDB 允许多个 NULL）。

### 普通索引

最基础的索引类型，没有任何唯一性约束。

### 前缀索引

对于**长字符串**字段（如 `VARCHAR(255)`, `TEXT`），如果对整个字段建索引太占空间，可以只取该字段的**前 N 个字符**建立索引。

## 字段个数

### 单列索引

只基于表中**某一个列**创建的索引。

### 联合索引

基于表中**两个或更多个列**组合创建的**一个**索引。

现在大致的理解了这些索引的特点，接下来我们从这些角度大致的说一些

# 按数据结构分类

| 索引类型      | InnoDB                                                     | MyiSAM | Memory |
| ------------- | ---------------------------------------------------------- | ------ | ------ |
| B+Tree索引    | Yes                                                        | Yes    | Yes    |
| HASH索引      | No（不支持hash索引，但是在内存结构中有一个自适应hash索引） | No     | Yes    |
| Full-Text索引 | Yes(MySQL 5.6版本后支持)                                   | Yes    | No     |

在创建表的时候，InnDB存储引擎会根据不同场景选择不同的列作为索引：

- 如果有主键，默认使用主键作为聚簇索引的索引键
- 如果没有主键，就选择第一个不包含NULL值的唯一列作为聚簇索引的索引键
- 如果上面两个都没有的情况下，InnoDB将会自动生成一个隐式自增id列作为聚簇索引的索引键

其他索引都属于辅助索引，也被称为二级索引或非聚簇索引。创建的主键索引和二级索引默认使用的是B+Tree索引

# 按物理存储分类

从物理存储的角度看，索引分为聚簇索引、二级索引

主键索引的B+Tree的叶子结点存放的是实际数据，所有完整的用户记录都存放在主键索引的B+Tree的叶子节点里

二级索引的B+Tree的叶子结点存放的是主键值，而不是实际值

# 按字段特性分类

从字段特性的角度来看，索引分为主键索引、唯一索引、普通索引、前缀索引

其实就是和上面的介绍差不多

# 按字段个数分类

从字段个数的角度来看，索引分为单列索引、联合索引

建立在单列上的索引称为单列索引，比如主键索引

通过将多个字段组合成一个索引，该索引就被称为联合索引
