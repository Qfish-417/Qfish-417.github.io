# 应用层

应用层有的协议有HTTP、HTTPS、CDN、DNS、FTP

## HTTP

### 请求报文

- 请求行：包含请求方法（如 `GET`, `POST`, `PUT`, `DELETE`）、请求目标（也就是灵路径，比如`/index.html`, `/api/login`）和HTTP协议版本
- 请求头部：包含关于请求的附加信息，比如申请的目标服务器的域名，客户端身份。
- 空行：请求头部和请求体之间用空行隔开
- 请求体：可选，包含请求的数据，通常用于POST请求等需要传输数据的情况

### 响应报文

- 状态行：包含HTTP协议版本、状态码和状态信息
- 响应头部：包含关于响应的附加信息，如Content-Type、Content-Length等
- 空行：响应头部和响应体之间用空行隔开
- 响应体：包含响应的数据通常是服务器返回的HTML、JSON等

### HTTP常用状态码



| 状态码 | 具体含义                                                 | 常见的状态码       |
| ------ | -------------------------------------------------------- | ------------------ |
| 1xx    | 提示信息，表示目前是协议处理的中间状态，还需要后续的操作 |                    |
| 2xx    | 成功，报文已收到并被正确处理                             | 200、204、206      |
| 3xx    | 重定向，资源位置发生变动，需要客户端重新发送请求         | 301、302、304      |
| 4xx    | 客户端错误，请求报文有误，服务器无法处理                 | 400、403、404      |
| 5xx    | 服务端错误，服务器在处理请求时内部发生了错误             | 500、501、502、503 |

状态码分为以上5大类

### HTTP层请求的类型

- GET：用于请求获取指定资源，通常用于获取数据
- POST：用于向服务器提交数据，通常用于提交表单数据或进行资源的创建
- PUT：用于向服务器更新指定资源，通常用于更新已存在的资源
- DELETE：用于请求服务器删除指定资源
- HEAD：类似GET请求，但只返回资源的头部信息，用于获取资源的元数据而不获取实际内容

### HTTP长连接

采用的是***请求-应答***的模式，也就是客户端发起了请求，服务端才会响应。

由于HTTP是基于TCP传输协议实现的，客户端与服务端要进行HTTP通信前需要先建立TCP连接，然后客户端发送HTTP请求，服务端收到后就返回响应。这就是***请求-应答***的模式。随后就会释放TCP连接。

如果每次请求都要经历：建立TCP-->请求资源-->响应资源-->释放连接，这就是HTTP短连接。

然而HTTP长连接不一样，它是使用同一个TCP连接来发送和接收多个HTTP请求/应答，避免连接建立和释放的开销。

### 端口号

源端口号：发送方应用程序的端口

目的端口：接收方服务的端口

作用：告诉操作系统这个操作系统是哪个程序发的的？要交给谁

### 序列号

这是 TCP **可靠传输**的核心。它不是包的编号，而是**本报文段中第一个字节的数据在整个字节流中的编号**。

### 确认号

只有当标志位中的 **ACK=1** 时，这个字段才有效。它表示：“**我已经成功收到了你发来的所有直到 `N-1` 的字节，我期望你下次从 `N` 开始发**

### 数据偏移

也叫“头部长度”。因为 TCP 头部有可选部分（Options），长度不固定。 告诉接收方：“数据部分从哪里开始？”

### 标志位

| 标志位  | 全称           | 含义与作用                                                   | 典型场景                                         |
| ------- | -------------- | ------------------------------------------------------------ | ------------------------------------------------ |
| **URG** | Urgent         | 紧急指针有效。告诉接收方这个包里有“加急”数据，要优先处理     | 极少使用，早期用于中断操作 (Ctrl+C)。            |
| **ACK** | Acknowledgment | 确认号有效。绝大多数数据包（除了第一次 SYN）都置为 1。       | 几乎所有通信过程。                               |
| **PSH** | Push           | 推送。告诉接收方：“别等了，缓冲区里的数据赶紧交给应用程序处理！” | 交互式应用（如 Telnet, SSH），希望打字立即显示。 |
| **RST** | Reset          | 重置连接。强制断开连接，通常是因为出错了（如端口未开放，或收到非法包）。 | 连接异常、拒绝连接。                             |
| **SYN** | Synchronize    | 同步序列号。**只在建立连接时用到**。                         | **三次握手**的第 1、2 步。                       |
| **FIN** | Finish         | 结束。发送方数据发完了，请求断开连接。                       | **四次挥手**的第 1、3 步。                       |

- 建连用 **SYN**
- 断连用 **FIN**
- 出错用 **RST**
- 平时确认用 **ACK**
- 加急用 **URG**, 催促用 **PSH**

### 窗口大小

**流量控制**的核心。告诉发送方：“我的接收缓冲区还剩多少空间，你最多还能发多少字节。”

### 校验和

用于检测头部和数据在传输过程中是否损坏。

### 紧急指针

只有当 **URG=1** 时才有效。

### 可变长度

用于扩展功能，不是必须的。

## TCP三次握手

### 第一次

开始时，客户端和服务端都处于CLOSE状态。服务端主动监听某个端口，处于LISTEN状态。

客户端会随机初始化序号，将此序号置于TCP首部的<u>***序号***</u>字段中，同时把**<u>SYN</u>**标志位置为1，表示SYN报文。接着把报文发送给服务端，表示向服务端发起连接，之后客户端处于SYN-SENT状态

### 第二次

服务端收到客户端的SYN报文后，首先服务端也随机初始化自己的序号，将此序号填入TCP首部的序号字段中，首次把TCP首部的确认应答号字段填入（客户端随机传递过来的序列号+1），接着把SYN和ACK标志位置为1。最后把报文发给客户端，之后服务端处于SYN-RCVD状态

### 第三次

客户端收到服务端报文后，还要向服务端回应最后一个应答报文，TCP首部ACK标志位置为1，其次应答号为（服务端随机传递过来的序列号+1），最后把报文发送给服务端，这次报文可以携带客户到服务端的数据，之后客户端处于ESTABLISHED状态

服务端收到客户端的应答报文后，也进入这个状态

这时候发现第三次握手是可以携带数据的，前两次握手是不可以携带数据的

1. **第三次握手可以携带数据**：因为此时连接在逻辑上已经**基本建立**，客户端确认了服务器的接收能力，发送数据是安全的。
2. **前两次握手通常不携带数据**：主要是为了**防止资源浪费**、**避免重传歧义**以及**遵循协议状态机逻辑**。虽然 RFC 标准没有绝对禁止第一次握手带数据，但在实际实现和最佳实践中是被严格禁止的。

### 为什么要三次握手

为什么要三次呢？而不能是两次、四次？

因为三次握手可以阻止重复历史连接的初始化（主要原因），同步双方的初始序列号，避免浪费资源

接下来详细解释一下

#### 避免历史连接

我们考虑一个场景，客户端发送一个SYN（seq=99），然后客户端出问题了，并且这个SYN报文还被阻塞，服务并没有收到，接着客户端重启，再次建立连接，发送SYN（seq=100）。（不是重传，SYN序列号不一样）。

这时候客户端收到的确认号是100而不是101，于是就会回RST报文（拒绝连接），服务端收到RST报文后就会释放连接，后续最新的SYN抵达服务端后就可以建立正常的三次握手

#### 同步双方初始序列号

通信双方都必须维护一个序列号，序列号是可靠传输的一个关键因素。

- 接收方可以去除重复的数据
- 接受方可以根据数据包的序列号按序接受
- 可以标识发送出去的数据包中，哪些是已经被对方收到的

#### 避免资源浪费

如果是两次握手，由于服务端不清楚客户端是否收到自己回复的ACK报文，服务端每收到一个SYN就只能先主动建立一个连接。这就会导致如果SYN报文在网络中阻塞，重复发送多次SYN报文，那么服务端在收到请求后就会建立多个冗余无效链接，造成不必要的资源浪费

## TCP四次挥手

如果说“三次握手”是两个人见面打招呼、确认彼此能听懂对方说话；那么“四次挥手”就是两个人**正式告别**。

为什么需要**四次**而不是三次？核心原因在于 **TCP 是全双工通信（Full-Duplex）**。

- **全双工**意味着：数据可以同时在两个方向传输（客户端→服务器，服务器→客户端）。
- **断开时**：必须**分别**关闭这两个方向的通道。
  - 先关闭“客户端→服务器”的通道。
  - 再关闭“服务器→客户端”的通道。
  - 这两个动作不能合并，因为一方说“我没话说了”（发送 FIN），不代表另一方也“没话说了”（可能还有数据要传完）。

#### 第一次

客户端发送一个 **`FIN`** (Finish) 报文给服务端。

这时候客户端从 `ESTABLISHED` 变为 **`FIN_WAIT_1`**。此时客户端进入“半关闭”状态：只收不发。

#### 第二次

服务端收到 `FIN` 后，立即回复一个 **`ACK`** 报文。

服务端从 `ESTABLISHED` 变为 **`CLOSE_WAIT`**，此时服务端进入“半关闭”状态

- 客户端→服务端：已关闭（客户端不再发数据）。
- 服务端→客户端：**依然通畅**！服务端还可以继续向客户端发送剩余的数据。

客户端收到 ACK 后，从 `FIN_WAIT_1` 变为 **`FIN_WAIT_2`**，客户端现在处于等待状态，等待服务端也发送 FIN。

#### 第三次

当服务端把所有剩余数据都发送完毕后，它发送一个 **`FIN`** 报文给客户端。

服务端从 `CLOSE_WAIT` 变为 **`LAST_ACK`** (Last Acknowledgment)。此时服务端也在等待最后的确认，以确保客户端收到了它的 FIN。

#### 第四次

客户端收到服务端的 `FIN` 后，回复一个 **`ACK`** 报文。

客户端从 `FIN_WAIT_2` 变为 **`TIME_WAIT`**。

**注意**：客户端**不会立即关闭**！它会启动一个计时器（通常是 2MSL，约 60-120 秒），停留在这个状态一段时间后才真正进入 `CLOSED`。

服务端收到 ACK 后，立即从 `LAST_ACK` 变为 **`CLOSED`**。服务端彻底释放连接资源。

那么问题来了，为什么还要进入**TIME_WAIT**？有两个原因：1、**保证最后一个 ACK 能到达服务端**。2、**让网络中残留的旧数据包消失**

#### 为什么4次握手中间两次不能变成一次呢？

服务端在收到客户端的FIN报文时，内核会马上返回一个ACK应答报文，但是服务端应用程序可能还有数据要发送，所以不能马上发送FIN报文，所以将FIN控制权交给服务端应用程序。所以第三次控制权不在内核，所以才分开发送。

当然还是有特殊情况的，当没有数据要发送，并且开启TCP延迟确认机制，那么第二和第三次挥手就会合并传输，可以出现三次挥手
